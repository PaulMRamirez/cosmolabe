# Goal: the device-truth pass (W4, runs the moment hardware is in hand)

This goal is drafted ahead of need so it preempts whatever session is next the moment a physical iPhone is available; it is the W4 item that cannot be simulated, and every number it produces is currently a recorded assumption somewhere (M-0007's platform facts, M-0001's deferred measurement legs, the ComputeProfile caps). Design 02 section 7 names the pass exactly: oldest supported iPhone plus current if both are in hand, and if only one is, record which and carry the other honestly rather than extrapolating.

Steps: first, the Companion prototype build: wrap the panel in native chrome via `pal-capacitor` (the two-shell thesis's lightweight surface; the workbench is explicitly not this build) and get it running on the device with the panel in 'transfer' mode, which is the mode the panel ships and by design the Companion baseline (M-0007 records that Capacitor cannot set the isolation headers on the initial document), so this pass tests the panel as-is with no new compute mode required. Second, the probes, recorded as observed fact with device model and OS version: WKWebView SharedArrayBuffer availability (does `crossOriginIsolated` report true under the Capacitor scheme handler, does the capacitor#6182 limitation still hold), and WebGPU presence and adapter identity inside WKWebView specifically (the iOS 26 default-on claim verified where the Companion would actually run, per the M-0007 caveat). Third, the sustained-load truth: a 20 minute scrub session against a loaded scene, recording thermal behavior (throttling onset, frame-time degradation curve, device temperature warnings) and the frame-time and memory measurements M-0001 deferred to exactly this pass (the spine acceptance's two open legs). Fourth, OPFS budget behavior: the kernel cache's real quota on device, eviction behavior, and whether the pack-min set survives a storage-pressure cycle. Fifth, after the measurements and from them, the `profiles` package, re-homed here at the Session 8 pre-merge gate as a post-measurement step (no longer a floating stretch; its carry count ended at four): the skeleton (the M-0007 tier ladder, the capability probe resolving RenderProfile and ComputeProfile) plus tier defaults derived from the numbers steps two through four measured, consumed by the panel's device-class defaults; the observed device lands in its honest tier rather than a hoped-for one.

Verified by: the Companion prototype runs the panel in 'transfer' mode on the physical device with all four product kinds materializing (photograph or screen recording as the artifact, since CI cannot observe a phone); the probe results, thermal curve, frame-time and memory numbers, and OPFS behavior recorded with device model and OS version; the M-0007 addendum written from those observations and M-0001's deferred legs closed or honestly re-deferred with what was measured; the profiles package skeleton with its tier defaults derived from this session's measured numbers (a post-measurement step, not a stretch; if the session ends before it, the carry is recorded with the data attached so the derivation needs no re-measurement); `pnpm verify` green locally and in CI; strict seam green; the sibling baseline compare green; the carryover ledger updated.

Exit: run scripts/review-on-return.sh and refresh the register and brief (M-0007 and M-0001 both carry review-on-return flags, and this session changes what Aaron needs to review on both); run /verify-spec and commit the report; draft or resume the next goal file (SESSION-8-MMGIS-EMBED if it was preempted, else the W4 gate evaluation).
