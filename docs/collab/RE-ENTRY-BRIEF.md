### Named finding: naked epoch strings parse as local time (Class B, review-on-return)

Epoch strings without a zone suffix, for example the scene `defaultTime: '2004-07-01T02:48:00'` in `packages/core/src/__tests__/_harness/scenes.ts`, go through JavaScript `Date` parsing in the cosmolabe path and therefore resolve in the machine's local timezone. Observed consequence: the committed goldens for `golden-fingerprints.test.ts` were captured on a Pacific machine, so a UTC runner computes Earth's Keplerian position about 738000 km away and the test fails; the first CI run caught exactly this. Workaround now in the gate: root `verify:cosmolabe` pins `TZ=America/Los_Angeles` for the cosmolabe test run, which reproduces the goldens deterministically on any machine without touching cosmolabe source. The real fix is a Class B change in your paths, landed with a baseline diff attached and flagged review-on-return: make epoch parsing zone-explicit (UTC) wherever catalog and test epochs are read, and regenerate the goldens under UTC. Until that lands, any new golden or baseline capture must run under the gate's pinned environment, and Session 2 records that environment alongside the baselines.
