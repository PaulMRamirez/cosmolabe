# ADR-0011: Native-first analysis engines; GMAT as an offline oracle, runtime backend deferred

Status: Accepted
Date: 2026-06-19

Same family as ADR-0008 (integrate by seam, not by embedding). Grounded in
docs/STK_PARITY_SPEC.md Section 8.

## Context

Surpassing STK requires deep analytical capability (high-precision propagation,
optimizing mission-control sequences, orbit determination). NASA GMAT (General
Mission Analysis Tool) is an Apache-2.0 mission-analysis application that already
implements much of this. The question is whether Bessel should depend on GMAT at
runtime to supply those engines. The binding constraints: Bessel is an offline-
capable web/WASM product (a browser-WASM port of GMAT is impractical given its
wxWidgets/C++ dependency set and size), the runtime is native-first, and every
analysis quantity must be validated against an independent reference.

## Decision

1. Native-first runtime. The analysis engines are implemented natively in
   TypeScript over CSPICE-WASM (the packages in docs/STK_PARITY_SPEC.md Section 10).
   There is no runtime dependency on GMAT; the offline PWA keeps the native engines.
2. GMAT is used as an offline validation oracle and an algorithm reference, not a
   runtime dependency: committed static fixtures (generated by GMAT out of band, not
   GMAT running in CI) cross-validate the native engines, and GMAT's published
   algorithms and test cases inform the implementations.
3. A runtime GMAT backend is deferred behind a decision gate. If a domain later
   justifies it (for example orbit determination, or full NxN HPOP parity), a GMAT
   backend would be added deliberately behind the PAL ComputeProvider seam,
   desktop or server only, never in the offline PWA, and recorded in a new ADR.

## Consequences

- The offline PWA model holds: no server or native runtime dependency is introduced
  for analysis.
- The native engines are validated against an independent, open reference without a
  CI dependency on a GMAT install.
- The seam exists to add a GMAT backend later without a rewrite, but adopting one is
  a gated, deliberate decision, not the default.
- Until then, the deepest STK-class capabilities (HPOP NxN with drag/SRP, an
  optimizing MCS, orbit determination) are native work items, tracked in
  docs/STK_PARITY_SPEC.md.
